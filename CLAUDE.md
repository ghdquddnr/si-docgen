# CLAUDE.md — si-docgen

## 프로젝트 개요

한국 SI 프로젝트의 표준 산출물(요구사항정의서, 테스트시나리오, 요건추적표, WBS, 화면정의서 등)을
LLM 기반으로 자동 생성하는 웹 애플리케이션.

- 입력: 요구사항정의서, RFP, 회의록 등의 원천 문서
- 출력: 고객사 양식이 그대로 보존된 MS Office 파일 (.docx / .xlsx / .pptx)
- 포지셔닝: 완전 자동화가 아닌 "AI 초안 생성 + 사람 검수 게이트" 모델

## 절대 원칙 (아키텍처 불변 규칙)

다음 원칙은 어떤 리팩토링·기능 추가에서도 위반 금지. 위반이 필요해 보이면 코드를 쓰지 말고 먼저 사용자에게 보고할 것.

1. **LLM은 절대 문서 파일을 직접 생성하지 않는다.**
   LLM의 역할은 구조화된 JSON 생성까지. 파일 렌더링은 결정론적 코드(렌더러)가 전담한다.
2. **렌더러는 순수 함수다.**
   `(검증된 Pydantic 모델, 템플릿 파일 경로) → 출력 파일`. 렌더러 내부에서 LLM 호출, 네트워크 I/O, 전역 상태 접근 금지.
3. **모든 LLM 출력은 Pydantic 스키마 검증을 통과해야 한다.**
   검증 실패 시 오류 내용을 포함해 최대 3회 재시도. 3회 실패 시 예외를 올리고 절대 미검증 데이터를 렌더러에 전달하지 않는다.
4. **템플릿은 보존하고, 값만 쓴다.**
   엑셀/워드/PPT 양식을 코드로 새로 그리지 않는다. 템플릿 파일을 열어 플레이스홀더 치환 또는 셀 값 주입만 수행한다. 서식(테두리, 병합, 결재란, 폰트)은 템플릿의 책임이다.
5. **LLM 호출은 추상화 계층을 통해서만 한다.**
   `backend/app/llm/` 의 클라이언트 래퍼(LiteLLM 기반)를 경유. 코드 곳곳에서 특정 벤더 SDK 직접 호출 금지. 모델명은 설정 파일에서만 관리.
6. **산출물 간 ID 추적성을 깨지 않는다.**
   요건 ID(REQ-xxx) → 화면 ID(SCR-xxx) → 테스트케이스 ID(TC-xxx)의 참조 관계는 스키마 레벨에서 강제하며, 생성 단계에서 존재하지 않는 ID 참조가 발견되면 검증 실패로 처리한다.

## 기술 스택

- **Backend**: Python 3.12+, FastAPI, Pydantic v2, LiteLLM
- **렌더링**: docxtpl(Word), openpyxl(Excel), python-pptx(PowerPoint), Playwright(화면 목업 HTML 렌더링 → PNG 캡처)
- **Frontend**: Next.js (App Router), TypeScript, Tailwind CSS
- **DB**: PostgreSQL (프로젝트/산출물/검수 이력 관리)
- **패키지 관리**: backend는 uv, frontend는 pnpm

## 디렉토리 구조

```
si-docgen/
├── CLAUDE.md
├── docs/
│   ├── PRD.md
│   ├── ARCHITECTURE.md
│   ├── TASKS.md               # 작업 목록과 상태. 모든 작업의 시작점
│   └── schemas/               # 산출물별 스키마 설계 문서 (사람이 읽는 명세)
├── backend/
│   ├── app/
│   │   ├── schemas/           # Pydantic 모델. 산출물 1종 = 파일 1개
│   │   ├── renderers/         # 순수 함수 렌더러. 산출물 1종 = 파일 1개
│   │   ├── llm/               # LiteLLM 래퍼, 프롬프트 템플릿, 검증-재시도 루프
│   │   ├── pipelines/         # 원천문서 → JSON → 파일까지의 오케스트레이션
│   │   └── api/               # FastAPI 라우터
│   └── templates/             # 산출물 양식 파일 (.docx/.xlsx/.pptx)
├── frontend/
└── tests/
    ├── unit/
    └── golden/                # 골든 파일 테스트: 고정 JSON 입력 → 기대 출력 비교
```

## 코딩 컨벤션

### Python
- 모든 함수에 타입 힌트 필수. `Any` 사용 시 주석으로 사유 명시
- Pydantic 모델 필드에는 한국어 `description` 필수 — 이 description이 LLM 프롬프트의 스키마 설명으로 재사용됨
- 예외는 `app/exceptions.py` 의 커스텀 예외 계층 사용 (`ValidationFailedError`, `RenderError`, `LLMError`)
- 포매터/린터: ruff (format + check)
- 로깅: 표준 logging, LLM 호출은 요청 모델명·토큰 수·소요 시간을 INFO 레벨로 기록. 프롬프트 본문은 DEBUG 레벨

### TypeScript / Next.js
- 서버 컴포넌트 우선, 클라이언트 컴포넌트는 `"use client"` 필요 시에만
- API 호출은 `frontend/lib/api.ts` 의 타입 정의된 클라이언트만 사용
- 상태 관리는 React 기본 훅으로 시작, 필요해지기 전에 외부 라이브러리 도입 금지

### 공통
- 주석과 문서는 한국어, 식별자(변수/함수/클래스)는 영어
- 산출물 도메인 용어 영문 표기 통일: 요구사항정의서=`requirement_spec`, 테스트시나리오=`test_scenario`, 요건추적표=`rtm`, 화면정의서=`screen_spec`, WBS=`wbs`

## 테스트 규율

- **렌더러는 골든 파일 테스트 필수.** `tests/golden/` 의 고정 JSON 입력으로 파일을 생성해 기대 산출물과 비교한다. 바이너리 비교가 아닌 내용 비교(openpyxl로 셀 값 추출, python-docx로 텍스트 추출 후 비교)로 구현한다.
- **스키마는 경계값 테스트 필수.** 빈 리스트, 최대 길이 초과, 잘못된 ID 참조가 검증에서 걸리는지 확인한다.
- LLM 호출이 포함된 테스트는 작성하지 않는다. LLM 계층은 모킹하고, 프롬프트 품질은 별도의 평가 스크립트(`scripts/eval/`)로 다룬다.
- 커밋 전 `pytest` 와 `ruff check` 통과 필수. 실패 상태로 커밋하지 않는다.

## 작업 흐름

1. 세션 시작 시 반드시 `docs/TASKS.md` 를 먼저 읽고 진행 중/다음 태스크를 파악한다.
2. 태스크 하나의 단위는 "커밋 가능한 최소 기능". 여러 태스크를 한 커밋에 섞지 않는다.
3. 태스크 완료 시: ① 테스트 통과 확인 → ② 커밋 → ③ `TASKS.md` 상태 갱신 + 다음 작업자가 알아야 할 특이사항 1~2줄 기록.
4. 커밋 메시지: `feat|fix|refactor|test|docs(scope): 한국어 요약` (예: `feat(renderer): 테스트시나리오 엑셀 렌더러 추가`)
5. 아키텍처 변경(새 디렉토리, 새 의존성, 절대 원칙과 충돌)이 필요하면 코드 작성 전에 사용자에게 선택지를 제시하고 승인을 받는다.
6. 사용자가 명시하지 않은 기능을 선제적으로 추가하지 않는다 (스코프 크리프 방지).

## 산출물 도메인 지식

각 산출물의 상세 스키마는 `docs/schemas/` 가 단일 진실 공급원이며, 아래는 최소 골격이다.

- **테스트시나리오(xlsx)**: TC ID / 연관 요건 ID / 대분류 / 중분류 / 시나리오명 / 사전조건 / 테스트 절차 / 기대 결과 / 결과(Pass·Fail) / 비고. 시트는 단위/통합 분리.
- **요건추적표 RTM(xlsx)**: 요건 ID / 요건명 / 화면 ID / 프로그램 ID / TC ID / 단계별 반영 여부. 테스트시나리오와 동시 생성하여 정합성 보장.
- **화면정의서(pptx)**: 슬라이드당 화면 1개. 화면 ID / 화면명 / 메뉴 경로 / 목업 이미지 / 항목 정의 표(번호·항목명·유형·필수·설명) / 처리 로직 설명. 목업은 LLM 생성 HTML을 Playwright로 캡처한 PNG를 삽입.
- **WBS(xlsx)**: 계층 번호(1.1.2 형식) / 태스크명 / 담당 역할 / 시작일 / 종료일 / 공수(MD) / 선행 태스크 / 산출물. 계층 번호와 일정 계산은 LLM이 아닌 렌더러 측 코드 로직으로 처리.

## 개발 로드맵 (현재 Phase 표시는 TASKS.md 참조)

- **Phase 0**: 렌더러 PoC — 양식 3종(테스트시나리오 xlsx, 요구사항정의서 docx, 화면정의서 pptx)을 하드코딩 JSON으로 채워 품질 검증
- **Phase 1**: 단일 파이프라인 — 원천 문서 업로드 → LLM JSON 생성 → 테스트시나리오 + RTM 출력 (CLI)
- **Phase 2**: 웹화 — FastAPI + Next.js, 업로드/진행 상태(SSE)/검수 화면(JSON 표 편집 후 재렌더링)/다운로드
- **Phase 3**: 다중 산출물 체이닝 + ID 추적성 + LiteLLM 기반 로컬/상용 모델 전환
- **Phase 4**: React Flow 기반 노드 캔버스 UI (역할별 에이전트 노드 구성)

## 금지 사항

- LLM 응답 텍스트를 정규식으로 파싱해 데이터 추출 (반드시 JSON 모드 + Pydantic)
- 렌더러 안에서의 LLM 호출 또는 외부 API 호출
- 템플릿 파일을 코드로 재생성하거나 서식을 코드로 직접 그리는 행위
- 테스트 없는 렌더러 머지
- `.env`, API 키, 고객사 실제 산출물 파일의 커밋 (templates/ 에는 자체 제작 양식만 허용)
